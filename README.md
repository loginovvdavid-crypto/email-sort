# Автоматизированная система обработки корпоративной почты

Система автоматически читает входящие письма IT-отдела, классифицирует их по категориям и раскладывает по папкам. Результат — структурированный архив писем и текстовый отчёт.

## Категории

| Категория | Описание |
|---|---|
| `incidents` | Критические сбои, массовые падения, URGENT |
| `access_requests` | Запросы доступа, права, новые сотрудники |
| `hardware` | Неисправность оборудования |
| `software` | Проблемы с ПО, приложениями |
| `documents` | Счета, акты, договоры |
| `hr` | Отпуска, больничные, графики |
| `monitoring` | Автоматические алерты систем мониторинга |
| `newsletters` | Дайджесты, рассылки, приглашения |
| `spam` | Спам и фишинг |
| `unknown` | Письма, не попавшие ни в одну категорию |

Категория `unknown` — страховочная: любое новое письмо, не подходящее под существующие правила, попадёт туда, а не потеряется.

## Требования

- Python 3.10+
- pytest (для тестов)

```bash
pip install pytest
```

## Запуск

```bash
# Стандартный запуск
bash run.sh

# С указанием папок
bash run.sh path/to/inbox path/to/output

# Напрямую через Python
python main.py inbox output
```

## Запуск тестов

```bash
pytest tests/ -v
```

## Структура проекта

```
email-sort/
├── src/
│   ├── email_reader.py      # Чтение и парсинг писем
│   ├── email_classifier.py  # Классификация по правилам
│   ├── email_sorter.py      # Перемещение файлов по папкам
│   └── report.py            # Генерация отчётов
├── tests/                   # Тесты
├── main.py                  # Точка входа
├── run.sh                   # Bash-скрипт запуска
└── inbox.zip                # Исходный набор писем
```

## Архитектура

```
inbox/*.txt
     │
     ▼
EmailReader        — парсит заголовки и тело письма
     │
     ▼
EmailClassifier    — правила по ключевым словам → категория
     │
     ▼
EmailSorter        — копирует файл в output/<категория>/
     │
     ▼
ReportGenerator    — report.txt + report.json
```

Классификация основана на keyword-правилах с приоритетами: спам проверяется первым, затем мониторинг и критические инциденты. Письма без совпадений попадают в `unknown`.
